## 6 · OPEN latent register (real code defects; not firing at today's seed scale — follow-up)

**Pattern A — client `.limit(N)` cap reused as an aggregate input (silent undercount at scale).**
A fetch cap sized for a *list* consumer is reused to compute a headline *count/sum*; undercounts
once a hive's data exceeds the cap. Instances (verified in code, latent at ≤30-row seed):
- `achievements.html` standings `.limit(500)` unordered → misranks a hive >~41 achievement-active members
- `ai-quality.html` `.limit(2000)` → "AI calls/spend (30d)" undercount for a busy hive
- `pm-scheduler.html` + `hive.html` scope items `.limit(500)` → overdue undercount at ~30+ assets × ~15 items
- `asset-hub.html` `.limit(200)` "total assets" tile · `alert-hub.html` `.limit(20)` total/critical
- (achievements weekXP was the same class — **fixed** in §3.)

## 7 · SYNTHESIS

**One root cause spans §1–§2 and Pattern A/B:** a known-good pattern applied **table-by-table /
consumer-by-consumer** but with **siblings skipped**. The security sweep hive-gated the tables it
touched and left `inventory_items` / `report_contacts` / `api_keys` / project-family behind; the
display code fixed `weekXP`-style caps where noticed but left the same `.limit()`-as-aggregate
shape in ≥6 other KPIs. The durable fixes are (a) **gate the class, not the instance** — the new
`validate_hive_write_isolation.py` now covers the write-hole siblings; a companion "limit-as-aggregate"
lint would cover Pattern A — and (b) **never derive a count/sum from a list-capped array; never
render a swallowed `{error}` as empty.**
