# Per-Page SaaS-Layer-Stack Bug-Hunt Roadmap

**Mandate (Ian, 2026-07-22):** the per-page 8-phase/v3 bug-hunt is COMPLETE (31 pages × 8 phases,
all gate-locked). Now hunt each page through the **13 full-stack SaaS PRODUCTION layers** — the
operational stack a real SaaS runs on, not just the request path. This is the SaaS-layer sibling of
`PER_PAGE_BUGHUNT_ROADMAP.md`: same method (live-confirm before claiming · gate every fix · ratchet a
per-page % per layer), new axis (production layers, not interaction phases).

**Canonical layer set** = `COMPREHENSIVE_STUDY_FULLSTACK_GATE.md` §2 (the 13 rows). This roadmap takes
that platform-wide layer taxonomy and drives it **per page**, so a page-specific operational bug
(this page double-fires the rate-limit bucket / never caches its heavy read / swallows its errors with
no log / doesn't degrade when the LLM chain is exhausted) can't hide behind a green platform average.

## §0 · ANTI-DRIFT DOCTRINE — read FIRST, every session, every sub-unit (Ian, 2026-07-22)

This arc is **138 open operational cells across 42 pages** — a long, multi-session grind exactly like the
per-page 8-phase hunt that reached 100% WITHOUT drift. It got there because the scoreboard + a ratchet gate
made "no drift" a **tested invariant, not a memory**. The same rails apply here. When a decision arises
("what's next / is this a bug / am I done / should I stop"), the answer is in these rails + the scoreboard —
**never** the tangent in front of you.

1. **THE SCOREBOARD IS THE COMPASS.** `PER_PAGE_SAAS_LAYER_SCOREBOARD.md` (generated by
   `tools/build_saas_layer_scoreboard.py`) is the SINGLE source of truth: every `(page × layer)` cell is
   **COVERED** (a registered gate locks it) · **OPEN** (to hunt) · **N/A** (no footprint). Before ANY
   hunting decision, read it. "What's next" = the highest-priority OPEN cell in the §4 execution order,
   full stop. ([[feedback_roadmap_percent_is_the_anti_drift_compass]])

2. **THE FORWARD RATCHET (the teeth).** The count of OPEN operational cells (C/RL/CA/L/AV) is a
   **forward-only CEILING** — it may only FALL. The registered gate **`saas-layer-scoreboard`**
   (`--check`) FAILs CI if it RISES: a new page, a new edge-fn, or a regressed gated cell that escapes
   coverage trips the drift-guard on first registration. Drift is now caught by CI, not by my vigilance.

3. **MEASURED-%, NEVER QUALITATIVE.** A cell reaches COVERED(100) ONLY when a registered `validate_*`
   gate locks it; verified-but-ungated caps at 75. Report the MEASURED open-cell count, never "looks
   handled." ([[feedback_measured_percent_not_qualitative_done]])

4. **REUSE FIRST — HUNT ONLY THE GAPS (the v3 §7.11 law).** F/D/AU/S are ALREADY gate-covered by the
   completed 8-phase hunt (page-battery · crud-rollback/read-battery · role-gate-server-backstop/isolation
   · XSS). **Re-confirm those, NEVER re-hunt them.** The ONLY new work is the ○ operational cells
   (C/RL/CA/L/AV). Re-probing a green cell is drift.

4b. **★ THE METHOD LAW — CENTRALIZE-FIRST, don't chip each page every time (Ian, 2026-07-22; the
    FAMILY_UFAI / FULLSTACK_COMPONENT_LIBRARY law).** *A defect that repeats on N surfaces is NEVER N bugs;
    it is ONE unadopted canonical component.* So for EVERY layer, BEFORE editing N pages, run the
    centralize-first test: **"is this ONE shared component (a helper / wrapper / global handler / edge-fn
    front-door) that every surface should adopt — or is it genuinely per-site?"** Build/adopt the CENTRAL
    component first; only what is *inherently* per-site (a caught error's local log call, a page's unique
    copy) gets chipped — and even then it routes through the central implementation so a future upgrade is
    ONE edit, not N. Worked example (Layer L, 2026-07-22): the 24 swallowed catches got their inherent
    per-site log, but the **capture backbone is centralized in `utils.js`** — `window.whLogError` (the ONE
    sink; add Sentry/aggregation there once → every page upgrades) + global `error`/`unhandledrejection`
    listeners that net the UNCAUGHT class platform-wide with ZERO per-page code. **The operational layers
    are mostly central by nature — prefer the central form:** RL → a shared 429-backoff wrapper on the
    db/fetch client (not per-page retry code) · CA → a shared cache layer (not per-page caching) · C → the
    `ai-gateway` front-door ALREADY centralizes the LLM fallback chain (adopt it, don't per-page it) · AV →
    a shared backend-down detector/banner (not per-page degraded handling). **Each layer's gate should
    verify ADOPTION of the central component, not N per-page fixes.** Chipping N pages when a central
    component would do is itself a form of drift.

5. **PER-CELL DEFINITION OF DONE (the flywheel — no skipped spoke):** find (live-probe the layer's
   failure mode) → fix → **verify LIVE** (confirm before claiming; a static finding is often a false
   positive) → **gate** (a registered `validate_*` with teeth + a `--selftest`) → **ratchet** the
   scoreboard cell COVERED → update the §7 ledger → NEXT. **A cell is NOT done until it is gated.**

6. **RECALL THE DISPOSITION + PROVE THE FAILURE before declaring a bug.** `memento_retrieve` the class
   and reproduce the failure scenario LIVE first. This session's proof: the assistant L-gap was real
   (error shown, never logged), but MANY signals are false positives — the Map.size class
   (`Set.delete()` read as a DB write), the modifier-class list-collapse (a homogeneous list read as 8
   panels), the `showToast('Write something first')` wording a narrow regex missed.
   ([[feedback_recall_the_disposition_before_declaring_a_bug]])

7. **A NEW DETECTOR'S HIGH FIRST-HIT COUNT = WIDEN/VERIFY, NOT MASS-FIX.** This session: the P4 gate
   flagged 12 → 11 were false; the double-submit gate flagged `Set.delete`; the arc-W grouping went RED on
   a measurement artifact. **Triage EVERY hit; key the detector on the PRECISE signal** (e.g. `db.` client
   for a write, base card-class for a list) before fixing anything.

8. **CONFIRM ON A FULL RUN BEFORE RATCHETING A CELL.** Never claim a layer gated-green from a partial /
   `--page` probe — a partial run can't see the page that regresses (arc-W went RED on the FULL sweep after
   a clean partial). Run the layer's gate across ALL its LIVE pages before ratcheting.

9. **STAY IN THE LAYER YOU'RE DRIVING (anti-tangent).** Drive ONE layer across its whole tier (e.g. **L**
   across all AI-heavy pages) to COVERED before switching layers — the §4 order decides, not the shiny
   finding. A cross-layer finding gets LOGGED to the scoreboard backlog, not chased mid-layer.

10. **READ THE WHOLE SCOREBOARD BEFORE "DONE."** One layer at 100% ≠ the arc done. Enumerate EVERY
    operational layer's open cells + the §4 tiers; the arc is done only when the OPEN count is 0 (or every
    residual is an explicit N/A with a reason). ([[feedback_measured_percent_not_qualitative_done]])

11. **MOMENTUM DISCIPLINE (arm `.momentum_drive`).** The turn-end test: a concrete OPEN cell in the
    scoreboard is a KNOWN next unit → ending is FORBIDDEN. Only (a) a fork Ian must decide · (b) a hard
    EXTERNAL ceiling (e.g. github unreachable for the push) · (c) an irreversible action that is the SOLE
    remaining item · (d) the OPEN queue genuinely EMPTY · (e) Ian says wrap THIS message — end a turn.
    Committing/pushing is Ian's standing gate: PIVOT to the next OPEN cell, never stop on it.

12. **THE NEXT QUEUE IS LOAD-BEARING + THE LEDGER IS THE MEMORY.** End every sub-unit by (a) regenerating
    the scoreboard, (b) appending a one-line §7 ledger entry (`page · layer · finding · fix · gate`), and
    (c) updating the `NEXT:` line. On resume, the scoreboard's OPEN cells + the NEXT line ARE the
    trajectory — trust them over any half-remembered context. The handoff/topic-memory carries them across
    session boundaries.

**One-line drift test, run at EVERY "should I do X?" moment:** *is X the highest-priority OPEN cell in the
scoreboard for the layer I'm currently driving?* If not, X is drift — log it and return to the cell.

## 2 · Scoring rubric (per page, per layer) — inherits `PER_PAGE_BUGHUNT_ROADMAP.md` §2

- **0** not started · **25** attempted/unverified · **50** happy-path verified live · **75** edge/negative
  verified · **100** all paths verified **AND locked by a registered gate** (verified-but-ungated caps 75).
- **Page-layer applicability**: a page owns a layer cell only if its footprint touches it (substrate card).
  A page with **no LLM call** auto-`N/A`s **C**; a **read-only** page auto-`N/A`s the write half of RL; a
  static/dev page `N/A`s most operational layers. N/A excluded from the denominator (same as v3).
- **Page % = mean of its LIVE layer cells. Roadmap % = mean of pages.** Report MEASURED %, never
  qualitative ([[feedback_measured_percent_not_qualitative_done]]).

## 3 · Method — reuse everything, hunt only the gaps (the v3 §7.11 law)

Most cells are ALREADY discharged by the completed 8-phase/v3 bug-hunt (F/D/AU/S) or by a standing
platform gate. This arc's job is to **map that coverage per page and hunt the ○/◐ operational gaps**, not
re-probe green cells. Per layer, the reuse:
- **A (edge contract)** — `validate_edge_fn_auth_gate` (caller-gate, done) + the substrate `Edge invokes:`
  per page → the NEW slice is per-page **error-shape + trace-id** on those calls.
- **C (LLM)** — `tools/ai_chain.py`/`_shared/ai-chain.ts` fallback chain → NEW: per page that invokes an
  LLM edge fn, prove graceful provider-exhaustion + a cache hit on repeat.
- **RL** — `checkAIRateLimit`/`checkUserRateLimit` → NEW: per page, prove a 429 is handled (backoff/toast,
  not a crash) + the page's AI + voice don't share one bucket to starvation.
- **CA** — SW shell + `ai_cache` → NEW: per page, prove the heavy read is cached + cache-bust applies.
- **L** — `window.onerror` + `logEvent` → NEW: per page, prove an error is captured, not swallowed silently.
- **AV** — `/health` + P7/P12 degraded → NEW: per page, prove fail-closed + honest empty-vs-error on
  backend-down (extends the P12 submit-path + P7 offline work).
