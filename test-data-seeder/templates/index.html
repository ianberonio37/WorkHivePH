📍 I made changes — what now? (the standard workflow)

  1. If schema or seeders changed → Reset then Seed Everything (Seed pane). Otherwise skip.
  2. Open Open pages panel → click the changed page (e.g. analytics-report.html) to test it against seeded data with one of the test logins.
  3. Run the gate that matches what you changed:
      • Schema / seeders changed → 🧪 Data Tests
      • Layout / rendering changed → 🎬 UI Tests (or 📷 Visual Regression for pixel-perfect)
      • Static code changes (no rendering) → 🚀 Release Gate
      • AI prompts / Groq logic → 🤖 Release Gate + AI Full
      • About to deploy to prod → 🚀 Mega Gate (everything)
  4. Bug found? → log in PRODUCTION_FIXES.md and tell Claude to fix.
  5. All clean? → tell Claude to commit + push.

Playbook — How to Build Any New Feature

Read this before clicking any Wizard or asking Claude to build a feature. Fuses the Skill-First rule, the Fuel/Engine/Brain wiring model, the registration checklist, and the hardening loop into one 7-step ordered workflow. Source-of-truth in memory: feedback_new_feature_playbook.md.

The 7-step ordered workflow
  1. Read the matching skills. Use the Skills Lookup table below. Read each SKILL.md before writing code. Scan for checklist items + anti-patterns.
  2. Find the canonical Engine source(s). Read from v_*_truth views, never raw Fuel tables. Use the Canonical-Source Lookup table below. If none fits, add a new view.
  3. Write the code on the right layer. Fuel = raw tables, Engine = canonical views + RPCs, Brain = orchestrators + AI, Dashboard = HTML, Driver = user. Never Dashboard→Fuel when an Engine view exists.
  4. Register on every surface. New page → LIVE_TOOL_PAGES, floating-ai, assistant prompt, nav-hub, 5 flow lists, Playwright spec. New edge fn → config.toml, deploy script, capability comment. New validator → run_platform_checks.py.
  5. Seed it. Add fixtures so the Tester can exercise it from a fresh reset. Run Data Tests (~5s) to confirm the seed is sane.
  6. Run the right gate. Default is NOT Mega Gate. Pick the lightest gate that covers what changed: Release Gate for architecture, +AI Full for prompts, UI Locks for forms, Visual for CSS, Mega Gate only pre-deploy.
  7. Decide on a new validator. Ask: "If this had broken silently, which gate would have caught it?" If none, extend an existing gate (preferred) or write a new one.

Seed data

First time? Click "Seed Everything". Individual buttons need hives/workers/assets to already exist.
When do I seed, and which buttons should I press?
First time / fresh DB: press Seed Everything. It runs all 9 modules in dependency order (~10k rows total, ~30-60s).

After a Reset: same — press Seed Everything.

Re-seed just one module: use the numbered buttons. They depend on earlier ones existing — e.g. you can't seed "4. Logbook" without "1. Hives & Workers" + "2. Assets" already in the DB.

You probably don't need to re-seed when: you're testing a UI/styling change, or a small data tweak. Just open the page and test.

You DO need to re-seed when: the schema migrated, the seeder logic changed, or your previous test corrupted state.

Danger zone

Irreversible.
What does Reset wipe, and when should I use it?
Wipes all seeded rows from your local Supabase: hives, workers, assets, PMs, logbook, inventory, skill matrix, marketplace, community. Auth users are preserved.

Use when: the schema changed (so old seeded rows have stale columns), seeder logic changed (so you want to regenerate from scratch), or your previous testing corrupted the data.

Safe? Yes — this only touches the local Supabase at 127.0.0.1:54321, never production. After Reset, click Seed Everything to repopulate.

Generate fake company CMMS data and download it as CSV. Phase 1 of the integration test harness.
What is this and how do I use it?
Simulates a Philippine manufacturing company's existing SAP PM, IBM Maximo, or generic CMMS database. Use it to design and test the WorkHive import wizard before any real client connects.

Import CSVs are in CMMS field format (SAP AUFNR, Maximo WONUM, etc.) -- feed these into the WorkHive Tier 1 import wizard to design your field mapping.

Patterns injected automatically: ~20% of assets have repeat breakdown history, ~15% of PM schedules are overdue, ~15% of parts are below reorder point.
