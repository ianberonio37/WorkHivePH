# WorkHive KPI Engine Map

Why this exists: four surfaces (Analytics Engine, Alert Hub, Asset Hub, Predictive
Maintenance) plus the Asset Brain AI all report on the same physical equipment,
but each one reads a different slice through the data. Users were seeing the same
metric label (MTBF, PM overdue, risk score) display different numbers across
pages and concluding the platform was wrong. It wasn't wrong, it was unlabelled.

This document is the labelling contract. Every surface declares **what it reads,
what window, and how often** so the driver knows which gauge they're looking at.

The metaphor:

| Layer | What it is | Where it lives |
|---|---|---|
| **Fuel** | Raw events | `logbook`, `pm_completions`, `inventory_transactions`, `asset_nodes` |
| **Engine** | Canonical computation | Postgres RPCs (`get_mtbf_by_machine`, ...) + canonical views (`v_*_truth`) + nightly batch (`batch-risk-scoring`) |
| **Brain** | Orchestration / decisions | `asset-brain-query`, `analytics-orchestrator`, `shift-planner-orchestrator`, `ai-gateway`, `floating-ai` |
| **Dashboard** | What the driver sees | `analytics.html`, `alert-hub.html`, `asset-hub.html`, `predictive.html` |
| **Driver** | The worker / supervisor | The human reading the page |

The Engine has **one canonical formula** per metric (e.g. MTBF = mean of
inter-arrival intervals of corrective entries within window W). Surfaces differ
only by which **window W** they ask for and how stale the answer is allowed to be.

## Per-metric canonical engine

| Metric | Canonical engine | Formula | Surfaces that read it |
|---|---|---|---|
| **MTBF** | `get_mtbf_by_machine` RPC (interactive) + `buildCompositeScoresV2` inline (batch). Same algorithm, different window. | Mean of inter-arrival intervals of `maintenance_type = 'Breakdown / Corrective'` entries within window W | Analytics (W = user period), Predictive/Asset Hub/Alert Hub (W = 365) |
| **MTTR** | `get_mttr_by_machine` RPC | Mean of `downtime_hours` for closed corrective jobs within window W | Analytics |
| **Failure frequency** | `get_failure_frequency` RPC | Count of corrective entries within window W | Analytics |
| **Downtime Pareto** | `get_downtime_pareto` RPC | Cumulative downtime ranked by machine | Analytics |
| **Risk score** | `batch-risk-scoring` rules-v2 composite scorer -> `v_risk_truth` | Weighted composite of 6 factors (PM overdue 0.30, fault frequency 0.30, time-to-MTBF 0.20, repeat fault 0.20, Weibull wear-out 0.15, FMEA top RPN 0.10), clamped to [0, 1] | Predictive, Asset Hub, Alert Hub, Asset Brain Query |
| **PM compliance / is_due** | `v_pm_compliance_truth` | `is_due = (last_anchor_date IS NULL) OR (last_anchor_date < now - 30 days)`; period-scoped counts at 30 / 90 / 365 / lifetime | Alert Hub; Analytics has its own period-scoped math via the orchestrator |
| **Asset 360** | `v_asset_truth` | asset_nodes + bridge columns (`legacy_asset_id`, `pm_asset_id`) + aggregate footprint (lifetime logbook entries, last failure, PM completed count, edge count) | Asset Hub, Asset Brain Query |
| **Failure timeline** | `v_logbook_truth` (canonical) joined via `legacy_asset_id` | Last N corrective + PM entries for the asset | Asset Hub, Asset Brain Query |

## Rules every new surface must follow

1. **Read the canonical view, not the underlying table.** If a `v_*_truth` exists
   for the concept, the surface reads it. Direct reads of `asset_risk_scores`,
   `pm_assets`, or raw `logbook` for these concepts are a validator failure
   (`validate_canonical_sources.py`).
2. **Display the window.** Every page that shows MTBF / risk / overdue / compliance
   must visibly state the window (e.g. "365-day failure window", "last 90 days",
   "30-day overdue floor") and the freshness (e.g. "Daily 13:00 PHT", "Live").
3. **Do not re-derive a metric the Engine already computes.** If `v_risk_truth`
   has `top_factors`, the surface (and any AI Brain talking about it) reads
   those factors. It does not recompute "why is this risky?" from raw history.
4. **Hive scoping on every query.** `.eq('hive_id', HIVE_ID)` is non-negotiable
   on every read (or `.is('hive_id', null)` for solo mode). See
   `validate_hive.py`.
5. **Asset identity flows through `asset_id` (uuid).** The bridge columns
   (`legacy_asset_id`, `pm_asset_id`) exist precisely so cross-table joins
   keep working without scattering name-string matches across the codebase.
   The `populate_asset_node_bridges` trigger (migration `20260512000004`)
   keeps the bridges populated for every new asset_node.

## Drift signals (watch for these)

- **Same label, different number across pages**: usually a window mismatch.
  Compare the on-page window chip on each surface. If both show the same
  window and still disagree, escalate to `validate_canonical_sources.py`.
- **Asset visible in one surface, missing in another**: usually a bridge gap.
  Check `asset_nodes.legacy_asset_id` and `asset_nodes.pm_asset_id` for that
  asset. The trigger should have filled them; if not, run the backfill
  migration again (it's idempotent).
- **Asset Brain narrative cites a factor not on the Predictive page**:
  Asset Brain should be reading `risk.top_factors` from `v_risk_truth` and
  quoting `explanation` verbatim. If it makes up a new factor name, the
  prompt's rule 3 was not followed; check `asset-brain-query/index.ts`
  composeContext output to make sure the risk block is actually populated.

## Remaining roadmap (priority-ordered)

### Near-term (small batch, fast wins)

**Phase 3.2 — Chip the remaining KPI-reading pages**
The chip coverage validator currently lists 6 pages on `KNOWN_NO_CHIP` debt
(`asset-hub.html`, `hive.html`, `shift-brain.html`, `dayplanner.html`,
`index.html`, plus the editor/utility pages). Each needs a chip placement
review:
- `hive.html` PM Health card and Open Work card -> per-card chip variant.
- `shift-brain.html` top-risk list -> chip below the page sub-header.
- `index.html` dashboard widgets -> per-widget chip pattern.
- `asset-hub.html` -> add a global page-level chip in addition to the per-card chip already on the Risk Profile section.
- `dayplanner.html` -> decide whether the logbook stream needs a chip or whether the page can be marked `chip-allow: chronological feed, not a KPI tile`.

When each page gets a chip, remove it from `KNOWN_NO_CHIP` in
`validate_kpi_chip_coverage.py`. The gate ratchets down automatically.

**Closing `KNOWN_DRIFT` entries**
`validate_canonical_sources.py` `KNOWN_DRIFT` currently lists ~14 entries.
Each is a documented migration target. Priority order (highest unlock first):
1. `ai-orchestrator/index.ts` reads `pm_assets` -> migrate hive-mode to `v_pm_compliance_truth` (pattern is identical to the analytics-orchestrator migration shipped in Phase 1.2; ~30 min).
2. `shift-planner-orchestrator/index.ts` reads `logbook` -> migrate to `v_logbook_truth` (~30 min).
3. `intelligence-report/index.ts` reads `logbook` 4 times -> migrate to `v_logbook_truth` (~45 min).
4. `failure-signature-scan/index.ts` reads `logbook` -> migrate to `v_logbook_truth` (~20 min).
5. `benchmark-compute/index.ts` reads `logbook` -> migrate to `v_logbook_truth` (~20 min).
6. `scheduled-agents/index.ts` reads `logbook` (4x) + `pm_assets` -> bulk sweep (~1 hour).
7. `embed-entry/index.ts` reads `pm_assets` -> migrate (~20 min).

Total estimated effort: ~3-4 hours to clear the entire backlog. Each
migration shrinks `KNOWN_DRIFT` and proves the canonical view scales.

### Medium-term

**Phase 2.1 — Route every AI call through `ai-gateway`**
Today the platform has multiple direct `callAI` importers (Asset Brain,
analytics-orchestrator's Groq synthesis, shift-planner-orchestrator,
ai-orchestrator, amc-orchestrator, voice-action-router, etc.). The
`ai-gateway` shipped 2026-05-11 already handles per-route quotas, PII
redaction, audit, and provider fallback; adoption is the remaining work.

Per-function changes are small (replace `callAI(prompt, opts)` with a
gateway-routed fetch that includes the route name and the existing opts),
but coordination is needed because each function's prompt + cache key +
audit category is different. Suggest one PR per function so the audit
trail is clean. Estimated 30 min per function * ~8 functions = ~4 hours.

**Phase 1.3 — `v_kpi_truth` materialised view (only if latency forces it)**
Today the canonical views are regular (live-recomputed) views. If an
Analytics phase ever exceeds 3 s p95 in production OR the Python API
times out at the 90 s cap, promote the relevant view to materialised
with a 1 h refresh cron. The view contract stays identical; only the
`freshness` field in `canonical_sources` changes. No surface change.

**Trigger to start**: a real production latency complaint or a
`platform-health` SLA breach. Do not pre-build.

### Long-term

**Phase 5 — Fuel cleanup (deferred until forced)**
The text/uuid bridges (`logbook.asset_ref_id` text, `inventory_items.linked_asset_ids` text[]) exist for backwards compat with the original asset model.
Migration to write the canonical UUID alongside, then deprecate the text
columns, is a 2-step cut-over touching every writer. Scope: ~2 sessions.
Defer until a real bug forces it (e.g. a text-vs-uuid join silently drops
rows). Today the `populate_asset_node_bridges` trigger keeps the bridges
populated so the existing scheme works.

## When the dashboard latency budget breaks

When live recomputation each refresh no longer fits, promote the relevant
`v_*_truth` view to a **materialized view** with a refresh schedule, and
update its `freshness` field in the `canonical_sources` registry. The view
contract stays the same; only the freshness SLA changes. Surfaces do not change.

Trigger to promote: a single Analytics phase taking > 3 s p95 in production,
or the Python API timing out at the 90 s cap.
