📖 WorkHive Platform Guardian — How to Use
What Each Button Does
▶ Validate Fast Runs all 40 checks in ~20 seconds. Use this after every code change. If green, you're safe to continue.
⚡ Auto-Fix When you see a red FAIL, click this. It tries to fix the problem automatically. Then run Validate Fast again to confirm.
🚀 Full Deploy Check Use only before pushing live. Checks if your real servers are responding. Takes ~60 seconds.
🧠 Self-Learn After fixing a bug, click this. It captures the lesson so the system gets smarter and won't miss it next time.
🔍 Improve Once a month. Searches the web for updates to engineering standards (NFPA, ISO, ASHRAE) that affect your calculators.
🔒 Gate Only Quick 3-second check: Is git clean? Are live servers up? Use before deploying if you already know the code is good.
📤 Commit & Push When Git shows WARN (uncommitted files), click this. Type your commit message, click OK. Done — no terminal needed.
What the Colors Mean
PASS — Green This check is working correctly. No action needed.
FAIL — Red Something broke. Fix it before deploying. Click Auto-Fix first, then Validate Fast again.
WARN — Yellow A known gap that is acceptable for now (e.g., Git has uncommitted files). Review but not always urgent.
SKIP — Blue This check was skipped because you're in fast mode. Normal. Run Full Deploy Check to include it.
Your Daily Routine
1Open dashboard → click Validate Fast → green = start coding safely.

3Ready to push live → click Full Deploy Check → green = safe to push.
4After fixing a bug → click Self-Learn → lesson captured for next time.

How It All Works (Simple Flow)
Think of it like a control room. Here is what happens behind the scenes when you use this dashboard.
1 Start Guardian.bat opens a small Python web server in the background, then opens this page in your browser. The server is the engine — nothing works without it.
↓
2 This dashboard talks to that server. When you click a button, the dashboard sends a request to the server asking it to run a specific script.
↓
3 The server runs the script and streams every line of output back to your terminal in real time. That is why you see the text appear live as it runs.
↓
4 40 validators run — each one checks a specific part of your platform. They cover calculations, data flows, AI quality, security, mobile UX, IoT readiness, enterprise integration, analytics, and more. See the "All 40 Validators" section below for plain-language descriptions.
↓
5 All results are saved to one report file platform_health.json. When a run finishes, the dashboard reads that file and updates all the cards you see here.
The Self-Healing Loop
FAIL found → Auto-Fix reads your skill rules and applies the fix → Validate Fast confirms it worked → Self-Learn writes the lesson to your skill files → next time the same problem appears, Auto-Fix already knows how to fix it automatically.
Who Does What
guardian_server.pyThe engine behind this web page. Routes your button clicks to scripts and streams output back.
run_platform_checks.pyThe boss script. Calls all 40 validators in order and writes the final report.
validate_*.py (40 files)The workers. Each one checks one area and returns PASS or FAIL.
platform_health.jsonThe report card. Written after every run. This dashboard reads it to show the cards.
learn.pyThe teacher. Reads failed runs and your git history, then writes lessons to your skill files.
autofix.pyThe mechanic. Reads lessons from skill files and applies fixes automatically.
platform_baseline.jsonSaved when all checks pass. Used to detect if something that was working suddenly breaks.
When Something Is Red — What to Do
Step 1: Read the terminal output — it tells you exactly which file has the problem.
Step 2: Click ⚡ Auto-Fix — it fixes known patterns automatically.
Step 3: Click ▶ Validate Fast — verify it turned green.
Step 4: Click 🧠 Self-Learn — capture the lesson.

If Auto-Fix doesn't help, the problem needs manual code review. Tell your developer which validator failed and what the terminal showed.
How to Read the Terminal Output
$ command A command that is running. Normal to see these.
PASS lines That check passed. All good.
FAIL lines That check found a problem. The line after it usually says which file and what's wrong.
✓ DONE Everything finished and passed. Safe to continue.
✗ DONE Something failed. Scroll up in the terminal to find the red FAIL line and read what it says.
💡 Tip: The terminal closes automatically when you click ✕ in its top-right corner. It stays open so you can read it — that's intentional.
