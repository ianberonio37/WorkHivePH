📖 WorkHive Platform Guardian — How to Use
What Each Button Does
▶ Validate Fast Runs all 45 validators across 5-6 check layers in ~40 seconds. Use after every code change. Skips live API calls (use Full Deploy Check for those). If all green with no red FAIL, you are safe to continue building.
⚡ Auto-Fix When you see a red FAIL, click this first. It reads your skill files to find the fix pattern and applies it automatically. Then click Validate Fast again to confirm it turned green. If it stays red, the fix needs manual review.
🚀 Full Deploy Check Run this before pushing live to production. It does everything Validate Fast does, PLUS calls your real Python API on Render.com and your real Supabase edge function to confirm they are responding. Takes ~60–90 seconds. Only needed at deploy time.
🧠 Self-Learn After fixing a bug, click this. It reads your git history and the validator output, then writes lessons to your skill files. Next time the same class of bug appears, Auto-Fix already knows how to handle it. Run once per bug fix session.
🔍 Improve Run once a month (or when you get a backlog alert). Searches the web for updates to 50+ engineering standards (NFPA, ISO, ASHRAE, PEC, ASME) and emerging technology signals (Groq models, pgvector, Supabase). Tells you what to update in your platform. Results appear in the Improvement Backlog panel below.
🔒 Gate Only Skips all 45 validators and just checks: Is git clean? Are live servers up? Is the backlog empty? Takes 3 seconds. Use this when you know your code is good and just want to confirm the environment is ready before pushing.
📤 Commit & Push When the Git card shows WARN (uncommitted files), click this. A dialog asks for your commit message — type it and click OK. The button commits all staged changes and pushes to your remote branch automatically. No terminal needed.
What the Colors Mean
PASS — Green This check is working correctly. No action needed. Safe to keep building.
FAIL — Red Something broke or a required pattern is missing. Fix it before deploying. Click Auto-Fix first, then Validate Fast again to confirm.
WARN — Yellow (Gate) A readiness issue that is not a code bug — for example, Git has uncommitted files, or the backlog has HIGH items. Review but not always urgent to fix immediately.
SKIP — Blue (in terminal) Inside a validator's output, SKIP means a known technical debt item — the check found a real gap but it is tracked as a backlog item rather than a blocking failure. The validator still shows PASS overall. Read the SKIP lines to understand your improvement backlog.
SKIP — Blue (Full Deploy) A live API check was skipped because you are in fast mode. Normal. Run Full Deploy Check to include it.

How It All Works (Simple Flow)
Think of it like a control room. Here is what happens behind the scenes when you use this dashboard.
1 Start Guardian.bat opens a small Python web server in the background, then opens this page in your browser. The server is the engine — nothing works without it.
↓
2 This dashboard talks to that server. When you click a button, the dashboard sends a request to the server asking it to run a specific script.
↓
3 The server runs the script and streams every line of output back to your terminal in real time. That is why you see the text appear live as it runs.
↓
4 45 validators run — each one checks a specific part of your platform across 4–6 layers. Layers 1–3 catch hard failures (FAIL). Layers 4–6 include improvement checks that show as SKIP (blue) in the terminal — these are known backlog items, not blocking failures. See the "All 45 Validators" section below for plain-language descriptions of every check.
↓
5 All results are saved to one report file platform_health.json. When a run finishes, the dashboard reads that file and updates all the cards you see here.
The Self-Healing Loop
FAIL found → Auto-Fix reads your skill rules and applies the fix → Validate Fast confirms it worked → Self-Learn writes the lesson to your skill files → next time the same problem appears, Auto-Fix already knows how to fix it automatically.
Who Does What
guardian_server.pyThe engine behind this web page. Routes your button clicks to scripts and streams output back.
run_platform_checks.pyThe boss script. Calls all 45 validators in order and writes the final report.
validate_*.py (44 files)The workers. Each checks one area across 4–6 layers. Hard failures exit 1 (red FAIL). Improvement checks exit 0 but show blue SKIP lines in the terminal.
platform_health.jsonThe report card. Written after every run. This dashboard reads it to show the cards.
learn.pyThe teacher. Reads failed runs and your git history, then writes lessons to your skill files.
autofix.pyThe mechanic. Reads lessons from skill files and applies fixes automatically.
platform_baseline.jsonSaved when all checks pass. Used to detect if something that was working suddenly breaks.
When Something Is Red — What to Do
Step 1: Read the terminal — find the red FAIL line. It names the file and exactly what is wrong.
Step 2: Click ⚡ Auto-Fix — it applies the fix automatically for known patterns.
Step 3: Click ▶ Validate Fast — confirm it turned green.
Step 4: Click 🧠 Self-Learn — write the lesson to your skill files.

If Auto-Fix doesn't help: The problem needs manual code review. The terminal tells you which validator failed, which file, and what line. Fix the code, then Validate Fast again.

Blue SKIP lines are NOT failures. They are known improvement items — read them when you have time, but they do not block you from deploying.
How to Read the Terminal Output
$ command A command that is running. Normal to see these.
PASS lines (green) That check layer passed. All good — nothing to do.
FAIL lines (red) That check layer found a problem. The line after it names the file, line number, and what is wrong. Fix it before deploying.
SKIP lines (blue) A known improvement or backlog item. The validator still passes (exit 0). Read these to understand what to improve — but they do not block you. Example: "assistant.html system prompt is 8327 chars — exceeds small model budget."
✓ DONE / READY Everything finished and all hard checks passed. Safe to continue building or deploy.
✗ BLOCKED Something failed. Scroll up to find the red FAIL line. Read exactly which validator and file is the problem.
💡 Tip: The terminal stays open after the run finishes so you can scroll and read. Click ✕ in its top-right corner to close it when done.
